# Colors

The cytario color system has **two tiers**. **Semantic tokens** (`--color-background`, `--color-primary`) express *intent* and adapt automatically across light and dark — build UI with these. **Primitives** (`--color-purple-700`, `--color-slate-100`) are the raw palette the semantic tokens are built from; treat them as reference, not values to use directly.

Every swatch is read live from the token stylesheets and resolved under both themes, so this page can't drift from the code.

cytario's identity is carried by purple and teal — the **Primary** (purple) and **Secondary** (teal) sections in the semantic tokens below. Token names follow the shadcn/ui convention (`background`/`foreground`, `primary`, `destructive`, …) so they map to clean, property-agnostic Tailwind utilities (`bg-background`, `text-destructive`).

---

## Semantic tokens

What you build with. Each token maps a purpose to a palette value, so code references intent rather than raw color. They remap per theme — use the **theme** toolbar (light / dark / side-by-side) to switch what's shown. Each card shows the primitive it currently resolves to (`→`).

## Primitives (palette)

The raw source palette: brand hues, the slate neutral ramp, status hues, plus base black/white and alpha scrims. These are theme-invariant — light and dark resolve identically — so each is shown once.

> **Primitives are not an API.** Using a primitive directly in a component (`--color-purple-700`, a hex literal) is a bug — it can't respond to theme and bypasses the semantic layer. Reach for a semantic token above; if none fits, that's a gap to raise with the design-system team.

## Accessibility and contrast

| Combination | Contrast ratio | WCAG AA (normal) | WCAG AA (large) |
| --- | --- | --- | --- |
| Purple 700 on white | 8.5:1 | Pass | Pass |
| Teal 500 on white | 2.8:1 | Fail | Fail |
| Teal 700 on white | 4.6:1 | Pass (large) | Pass |
| White on purple 700 | 8.5:1 | Pass | Pass |
| Slate 900 on white | 17.9:1 | Pass | Pass |
| Slate 600 on white | 6.7:1 | Pass | Pass |

**Key rule:** Teal-500 does not meet WCAG AA contrast on white backgrounds for normal text. Use **teal-700** or darker for text. Teal-500 is suitable for large text, icons, and non-text decorative elements.

**Avoid washed-out teal on solid fills.** A mid-teal fill (teal-600 / teal-700) paired with white text is the only way to hold AA contrast with a light label — but it reads muted and washed out, draining the brand vibrancy. Don't reach for it. For a solid teal surface (buttons, badges), use a **light teal fill (teal-300) with dark ink (teal-950)** instead: it keeps the teal lively *and* clears AA (8.6:1). This is exactly how the **Secondary** button is built. Reserve vivid teal-500 for accents, icons, and focus rings where no foreground text sits on top.

## Brand governance

**Approved palette:** The purple and teal brand colors were established as part of the cytario corporate design approved on 05.11.2024. Do not introduce new brand colors without approval.

**Color in regulatory contexts:** Regulatory submissions and labels should not rely on color alone to convey meaning (per IEC 62366-1 usability requirements). When color is used in the cytario application UI for diagnostic significance (e.g., annotations, heatmaps), ensure an alternative non-color indicator is also present.
